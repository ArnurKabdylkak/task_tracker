# Таск-трекер бот для Telegram

Бот для ежедневной отчётности по задачам в команде.

## Возможности

- Сотрудник отправляет задачи на сегодня (списком — каждая строка отдельная задача, или через `/add`).
- Кнопки под планом: отметить выполненной ✅ / вернуть в работу ↩️ / удалить 🗑.
- `/today` — план на сегодня, `/yesterday` — вчерашний план, `/week` — план на неделю.
- Начальник видит задачи всех сотрудников: `/all` (сегодня), `/all вчера`, `/all 2006-01-02`, `/team` — список сотрудников.
- `/pending` — кто из сотрудников ещё не отписался сегодня.
- `/export` — выгрузка отчёта в CSV (открывается в Excel): `/export`, `/export вчера`, `/export неделя`, `/export 2006-01-02`, `/export 2006-01-02 2006-01-07`.
- Утром (по умолчанию 10:00) всем сотрудникам приходит напоминание отписаться по задачам.
- Днём (по умолчанию 14:00) повторное напоминание уходит только тем, кто ещё не отписался, а начальнику — сводка по таким сотрудникам.

## Установка

Нужен Go 1.21+. Драйвер SQLite требует CGO (компилятор C, например gcc).

```bash
go mod tidy
go build -o tasktracker ./cmd/tasktracker
```

## Запуск

Получите токен у [@BotFather](https://t.me/BotFather), затем:

```bash
export BOT_TOKEN="ваш_токен"
export BOSS_IDS="123456789"      # ID начальника(ов) через запятую
export BOT_TZ="Asia/Almaty"
export REMINDER_TIME="10:00"
./tasktracker
```

Все переменные окружения описаны в `.env.example`.

## Запуск в Docker

Образ собирается multi-stage (CGO нужен для SQLite, бинарник линкуется
статически, финальный образ ~17 МБ на Alpine). База лежит в volume и
переживает пересоздание контейнера.

```bash
cp .env.example .env   # заполните BOT_TOKEN и BOSS_IDS
docker compose up -d --build
docker compose logs -f
```

Остановить — `docker compose down` (volume с БД сохраняется).

Без compose:

```bash
docker build -t tasktracker .
docker run -d --name tasktracker --restart unless-stopped \
  --env-file .env -v tasktracker-data:/data tasktracker
```

### Как назначить начальника

В `BOSS_IDS` можно перечислять через запятую как числовые **Telegram ID**, так и
**@username** — вперемешку, например: `123456789,@ivan,olga`.

- **ID** надёжнее: он не меняется. Узнать свой — команда `/id` у бота.
- **@username** удобнее читать, но при смене ника пользователь потеряет права
  начальника, а у кого-то ника может не быть вовсе. Сравнение регистронезависимо,
  префикс `@` необязателен.

Роль обновляется при следующем сообщении пользователя после перезапуска бота.

## Как пользоваться

1. Каждый сотрудник пишет боту `/start`.
2. Утром отправляет задачи списком, например:
   ```
   Доделать отчёт
   Созвон с клиентом в 15:00
   Ревью PR
   ```
3. В течение дня отмечает выполненное кнопками.
4. Начальник смотрит сводку командой `/all`.

## Структура

Проект разбит на слои. `internal/domain` — ядро (модели и интерфейс
`Repository`), от него зависят остальные пакеты, а оно — ни от кого.

```
cmd/tasktracker/main.go   — точка входа: сборка зависимостей и цикл обновлений
internal/
  config/                 — чтение настроек из окружения
  domain/                 — модели (User, Task) и контракт Repository
  storage/                — реализация Repository поверх SQLite
  bot/
    handler.go            — роутинг апдейтов, базовые хендлеры
    commands.go           — команды (в т.ч. логика прав начальника)
    render.go             — слой представления: тексты и клавиатуры
    sender.go             — интерфейс отправки и нарезка длинных сообщений
  scheduler/              — фоновые напоминания (утреннее + не отписавшимся)
  report/                 — формирование CSV-отчёта
  timeutil/               — даты и их русская локализация
```

Хендлеры и планировщик зависят от интерфейсов (`domain.Repository`, `Sender`),
а не от конкретных `*Store`/`*tgbotapi.BotAPI` — поэтому покрыты юнит-тестами
с моками (`internal/bot/handler_test.go`, `internal/storage/sqlite_test.go`).

```bash
go test ./...
```

## Заметки

- Данные хранятся в файле `tasks.db` (SQLite). Бэкап — просто копия файла.
- Напоминание уходит каждый день. Чтобы пропускать выходные — добавьте проверку
  дня недели в `Scheduler.sendMorning` / `sendFollowup` (`internal/scheduler`).
- Сменить хранилище (напр. на Postgres) можно, реализовав `domain.Repository`
  в новом пакете — остальной код менять не придётся.
- Для запуска как сервиса используйте systemd / Docker.
